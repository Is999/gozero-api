# 开发扩展指南

本文档用于新增 `gozero-api` 前台业务能力时快速对齐代码、文档、缓存和测试边界。

## 新增 HTTP 模块

1. 在 `internal/handler/routes.go` 增加 `NewXxxRouteModule`，实际路由拆到 `routes_xxx.go`。
2. 在 `internal/bootstrap/registrations.go` 的 `defaultRouteModules` 和 `DefaultRegistrationManifest` 同步新增模块。
3. 在 `internal/handler/route_contract.go` 补 `DefaultRouteContracts`，声明 method、path、RouteMeta 和接口文档路径。
4. 在 `internal/handler/route_meta.go` 补 `RouteMeta`，别名使用 `domain.resource.action`，访问边界必须声明为 `public`、`auth` 或 `internal`。
5. 在 `internal/handler/route_security_contract.go` 补 `DefaultRouteSecurityContracts`，安全链路只能选 `none`、`public`、`auth` 或 `internal`。
6. 需要登录态的路由必须走 `AuthMiddleware`；公开路由必须写清 token 校验边界。
7. 内网运维路由必须使用 `internal` 安全链路，并在 `RouteSecurityPolicies` 中显式空策略，避免前台签名加密规则误拦运维调用。
8. 同步补 `docs/site/接口文档`，格式遵循接口文档统一规范；测试会校验文档包含真实路由路径。
9. 新增或修改安全字段后，通过 `DefaultRouteSecurityManifest()` 复核 alias、method、path、chain 和字段策略，再同步 `docs/site/route_security_manifest.json` 和前端同名 route policy。

## 签名与加密轻量策略

- `RouteSecurityPolicies` 必须按路由显式声明字段级策略；未知路由不默认参与签名或加密。
- `RequestSign` 和 `ResponseSign` 只列关键字段，禁止使用 `*` 对请求体、响应 `data`、大结构体或分页列表整体签名。
- 请求加密头 `X-Cipher` 只能声明 `RequestCipher` 允许解密的字段列表，禁止使用 `cipher` 整包加密；响应 `ResponseCipher` 只能声明 `data` 下的字段路径，禁止使用 `cipher` 整包加密。
- 单个接口参与签名或加密的字段最多 8 个；普通字段最大 4KB，`json:` 小对象最大 8KB。
- 安全中间件读取请求体最多 64KB；更大的业务负载必须绕开安全中间件解析路径，走专用上传、异步任务或拆分字段。
- 字段路径优先覆盖 token、金额、订单号、手机号、邮箱、二次校验字段等重要数据；普通展示字段、大列表和统计集合不默认签名或加密。
- 使用 `json:` 前缀前必须确认字段值很小且确实需要整体序列化；大对象、大数组和分页结果不得使用 `json:` 做整体加密。
- 新增或修改安全策略时必须同步接口文档 `### 安全字段` 表，写清请求签名、请求加密、响应签名和响应加密字段；没有敏感字段时明确说明不参与签名或加密。
- 安全链路失败必须使用细分业务码，禁止把 AppID、秘钥、验签、解密、响应加密和安全字段超限全部退回通用认证失败。
- 前端安全策略以 `internal/handler/route_security_manifest.go:DefaultRouteSecurityManifest` 为同步依据，字段名称和顺序必须与后端清单一致；不在清单内的字段不得由前端自行加入签名或加密。
- `docs/site/route_security_manifest.json` 是前台工程同步快照，未知路由必须返回空策略，禁止默认全量签名或整包加密；同步流程见 `前端安全清单同步.md`。
- 修改签名串、`X-Cipher` 编码或字段数量边界时，必须同步 `internal/security/testdata/security_vectors.json` 和前端同名测试向量。

## 运行期配置

- 启动期配置会执行基础校验：`jwt_secret` 不能过短，Redis 地址和连接池必须有效，强制 Redis Collector 时必须配置 Stream。
- `Mode=pro/prod/production` 会执行生产硬校验：`jwt_secret` 和 `ops.config_reload_token` 不能使用占位值，运维令牌必须达到最小长度，登录限流必须启用，Redis TLS 不能跳过证书校验。
- 生产环境开放注册时必须启用 `auth.register_rate_limit`；所有启用的认证限流参数必须控制在启动校验允许范围内，避免超大窗口或超长锁定时间。
- `collector.redis.enabled=true` 时必须配置 `collector.redis.stream`，即使 `collector.transport=auto` 也不能保留半截 Redis Stream 配置。
- `security.secret_key` 未配置时兼容普通请求；一旦配置，启动期会校验 `app_id`、版本唯一性、稳定/灰度版本引用、链路开关和 AES/RSA 材料。
- `sign_status`、`crypto_status` 只能为 `0` 或 `1`；启用签名或加密时必须配置用户 RSA 公钥和服务端 RSA 私钥，启用加密时还必须配置 AES KEY 与 IV。
- `security.secret_key` 的同一材料字段只能配置明文或 `*_ref` 之一；生产环境启用灰度秘钥时必须配置 `gray_salt`。
- 业务读取运行期配置优先声明 `SysConfigKey`，再使用 `SysConfigLogic.GetBool`、`GetInt`、`GetString`、`GetFloat`、`GetObject` 或 `GetArray` 类型化读取。
- `GetCachedValue(uuid)` 仅用于兼容旧调用链或确实需要动态 uuid 的场景。
- `sys_config` 表名固定无 `api_` 前缀，Redis Key 使用 `common/rediskeys` 中的 `api:sys_config:%s:%s` 模板。
- 缓存缺失由 redsync 分布式锁保护回源，业务侧不要绕过逻辑层直接拼 Redis Key 或自行查表。
- 配置变更后的缓存刷新优先使用 `RenewByKey(key)`，兼容场景可使用 `RenewByUUID(uuid)`；不要在线扫描高基数 Redis Key。
- 配置热加载 HTTP 接口属于内网运维能力，只注册 `/internal/system/...` 路由，必须携带 `X-Ops-Token`，生产环境需要配置 `ops.config_reload_token`。
- `ops.config_reload_allowed_ips` 只能在内网来源上进一步收窄访问范围，不能用于放行公网 IP。
- `ops.config_reload_allowed_ips` 配置非法 IP/CIDR 或公网 IP/CIDR 时会启动失败。
- 生产网关或 Ingress 必须拒绝公网转发 `/internal/` 路径；应用层会拒绝转发头中的公网来源。

## 认证与限流

- JWT 必须包含 `jti` 和当前服务 `app_id`；鉴权时会校验 token、Redis session、用户状态和 AppID 一致性。
- Redis session TTL 不超过 JWT 过期时间；刷新 token 会创建新 `jti`、写入用户 jti 索引并删除旧 session，旧 session 删除失败时必须回滚新 session。
- 鉴权成功会 best-effort 懒补用户 jti 索引，兼容上线前已存在但尚未写索引的 session。
- 退出登录只删除当前 token 对应的 session；禁用用户、重置密码或风控封禁时使用 `AuthLogic.InvalidateUserSessions` 按 ZSET 索引精确失效全部 session。
- 登录入口使用 `auth.login_rate_limit` 按 IP 和用户名两个维度计数。
- 注册入口使用 `auth.register_rate_limit` 按 IP 计数，生产示例默认关闭注册。
- 注册、登录、刷新、退出、登录态鉴权失败、限流和用户全量失效会 best-effort 投递 `auth.security` 认证风控事件。
- 认证风控事件只包含用户 ID、脱敏用户名/IP/session 哈希、trace、route、app_id、动作和原因，禁止写入密码、token、明文用户名或明文 IP。
- 限流 Redis Key 必须继续使用 `common/rediskeys` 中的精确模板，不允许用通配扫描清理。

## Collector 轻量模式

- 前台 API 仅保留轻量 Collector：`sync` 适合本进程直接处理，`redis` 适合写入 Redis Stream。
- 新增 bizType 时通过 `RegisterProcessor` 或 `RegisterProcessorFunc` 注册 Processor，并保证 payload 是合法 JSON。
- 内置认证风控事件 bizType 为 `auth.security`，启动期会注册默认 Processor 统计 `gozero_api_auth_security_events_total{app_id,action,reason,category}` 指标；告警口径见 `认证安全指标与告警.md`。
- `collector.transport=sync` 或 Redis Stream 不可用回退 sync 时，默认 Processor 只做本进程指标聚合；`redis` 载体优先写入 Stream 供外部风控消费。
- Collector 关闭时不影响认证接口响应；事件 payload 不包含密码、token、明文用户名或明文 IP。
- 本项目不默认引入后台 taskqueue、DB outbox、Kafka 可靠投递和管理端重试界面；确需升级时必须重新评估可靠性、补偿和运维成本。

## 组件生命周期

- MySQL、Redis 等启动期核心组件统一注册到 `svc.ComponentRegistry`。
- `/api/ready` 只遍历组件注册表执行探测；新增核心依赖时不要在 `HealthLogic` 里临时追加检查分支。
- 服务停止时按组件注册顺序反向释放资源；新增组件必须同时评估 `Check` 和 `Close` 是否需要配置。

## 数据与缓存命名

- 前台业务表默认使用 `api_` 前缀，例如 `api_user`。
- 全局运行期配置只使用 `sys_config`，不得新增 `api_sys_config`。
- Redis Key 必须集中定义在 `common/rediskeys`，禁止业务代码临时拼高基数通配 Key。
- 涉及大表或高频查询时，必须先确认索引、批量边界、超时和库压力。

## 必补检查

- 新增默认路由或运行时入口后执行 `go test ./internal/bootstrap ./internal/handler`。
- 新增或修改签名、加密字段后执行 `go test ./internal/handler`，确认路由安全清单、接口文档和字段级策略一致。
- 更新前端安全策略时先比对 `docs/site/route_security_manifest.json`，未知路由不得回退到 `*` 或 `cipher`。
- 修改签名串或 `X-Cipher` 编码后执行 `go test ./internal/security`，并在前端执行对应 security vector 单测。
- 涉及 sys_config、缓存锁或 Collector 时执行对应包测试。
- 后端改动完成后优先执行 `go test ./...`，最后执行 `git diff --check`。
