# AI开发提示词

下面提示词可直接复制给 AI，用于功能开发、问题修复、重构和文档补全。

```text
你现在参与 api 项目的生产级开发与重构。请严格按以下规范执行，不能只给方案，除非我明确要求先不改代码。

一、工作方式
1. 修改前必须先阅读现有目录结构、代码风格、命名习惯和调用链。
2. 保持现有分层和风格，不引入无必要的新框架、新目录、新依赖或新抽象。
3. 改动必须小步、聚焦、可回滚，不顺手重构无关代码。
4. 重构必须保证功能不遗漏，并在交付说明中写清影响范围、迁移入口、验证方式，以及是否需要数据迁移、重刷或补偿任务。
5. 注释、配置和文档必须简洁清晰，只说明用途、取值边界、默认行为或风险点，不写冗长背景。
6. 配置热加载必须区分运行参数和启动期注册开关；路由、handler、插件、基础设施连接等启动期能力不能在线补注册或重建，需通过 restartRequired/restartReason 提示重启。
7. 新增默认路由或轻量运行时扩展必须同步 `组件注册清单.md`、`开发扩展指南.md` 和 `internal/bootstrap/registrations.go`。

二、代码规则
1. 命名短而准确，贴合业务语境；禁止随意命名、过度缩写、超长描述名和含糊的 `DefaultValue`。
2. 错误必须使用 `github.com/Is999/go-utils/errors` 包装。
3. 同一个函数里，同一个 `error` 值只包装一次；跨函数边界可以继续包装，因为那是新的调用语义和上下文，避免裸错误漏出去。
4. 业务中间层只返回错误，不打印 error 级日志；顶层 handler、启动入口、健康检查或不可返回错误的兜底协程统一打印 error，同一错误链路只允许最外层打印一次。
5. 错误上下文和任务日志必须带可追踪字段，例如 `workflow_id`、`mode`、`node`、`shard`、`uid`、`task_id`。
6. 常量按领域前缀集中定义，表达稳定业务含义或枚举语义；禁止魔法字符串。
7. 结构体使用名词或领域对象命名；请求以 `Req` 结尾，响应以 `Resp` 结尾，列表项以 `Item` 结尾。
8. 接口按能力命名，优先使用 `Reader`、`Writer`、`Repository`、`Provider` 等自然后缀；只有一个实现且无替换点时不要强行抽接口。
9. Redis Key 必须集中定义，禁止在业务代码临时拼接高基数通配 Key。
10. 目录布局必须符合当前项目分层，保持 `handler` / `logic` / `model` / `svc` / `types` 等边界清晰。
11. 前台业务表默认使用 `api_` 前缀；运行期系统配置表固定使用 `sys_config`，不得新增 `api_sys_config`。
12. 业务读取运行期系统配置优先声明 `SysConfigKey` 并使用类型化 getter，避免散落字符串 uuid 和类型断言。

三、中文注释
1. 新增或重构的变量、常量、方法、结构体、字段、map 字段、路由、GORM Model、接口和关键逻辑必须补中文注释。
2. 代码注释、配置注释和文档说明必须简洁清晰，不能啰嗦，也不能含糊。一句话能说明白的不要写成长段，注释只说明字段用途、取值边界、默认行为或风险点关键信息，不重复字段名，不写背景故事。
3. 顶层稳定契约中的 map、数组、切片等字段必须说明元素 key/value 或元素对象含义、来源、边界条件、排序/去重和空值语义。

四、数据库与缓存
1. MySQL 读写优先使用 GORM 链式调用；事务内必须沿用同一个 `tx`。
2. 原生 SQL / Lua 必须作为 `*.sql.tmpl` 或 `.lua` 代码资产，通过 `go:embed` 加载。
3. SQL / Lua 调用前必须通过 `embedasset.StripLeadingLineComments` 或当前包 helper 剥离文件头注释。
4. 禁止在 Go 文件中内联多行 SQL / Lua。
5. 禁止在线接口无界扫描高基数 Redis Key；必须使用精确 Key、索引集合、白名单模板或异步任务。

五、错误处理
1. 错误必须使用 github.com/Is999/go-utils/errors 包装。
2. 同一个函数里，同一个 error 值只包装一次；跨函数边界可以继续包装，因为那是新的调用语义和上下文，避免裸错误漏出去。
3. 业务中间层只返回错误，不打印 error 级日志。
4. 顶层 handler、启动入口、健康检查或不可返回错误的兜底协程统一打印 error 级日志，同一错误链路只允许最外层打印一次。
5. 错误上下文必须带 workflow_id、mode、node、shard、uid、task_id 等可追踪字段。

六、日志规范
1. 日志统一使用项目现有 loggerx/logx 风格。
2. 正常流程只打印关键节点，不刷屏。
3. 任务系统、收集器和批处理日志必须带 workflow_id、mode、node、shard 等链路字段。
4. 日志字段类型必须稳定；时间窗口和间隔字段要带单位后缀，例如 window_start_unix、interval_seconds。

七、接口与业务码
1. API 响应必须统一为 status/code/message/data。
2. 不同业务错误必须使用不同业务码；message 不能暴露 SQL、密钥、内部表名或真实下游错误。
3. 新增业务码必须补中文和英文文案。
4. 新增接口必须补接口文档，格式遵循 docs/site/接口文档/接口文档统一规范.md。
5. 新增路由必须补 RouteMeta、RouteSecurityContract 和中文说明；登录态路由必须走 AuthMiddleware；公开路由必须写清公开原因和 token 校验边界。
6. 修改登录、刷新、退出或鉴权逻辑时，必须同步校验 jti、Redis session、用户状态、AppID、用户会话索引和 `auth.security` 脱敏事件边界，并补生命周期测试。
7. 请求和响应签名只能在 RouteSecurityPolicy 中显式列出关键字段，禁止默认使用 `*` 对完整请求体、完整响应 data、大结构体或分页列表整体签名。
8. 请求和响应加密只能使用字段级路径，请求字段必须由 RouteSecurityPolicy.RequestCipher 声明，禁止默认使用 `cipher` 整包加密；只对 token、金额、订单号、手机号、邮箱、二次校验字段等重要字段做轻量处理。
9. 单个接口参与签名或加密的字段最多 8 个；普通安全字段最大 4KB，`json:` 小对象最大 8KB，禁止用安全链路承载大对象、大数组或分页列表。
10. 安全中间件读取请求体最多 64KB；超过该上限的业务数据不得依赖签名/加密中间件解析，必须拆成受控字段或走专用上传/异步流程。
11. 安全链路失败必须使用细分业务码，不能把 AppID、秘钥、验签、解密、响应处理和安全字段超限统一退回通用认证失败。

八、大数据查询与批处理
1. 涉及大表、大结果集或高频查询时，必须先评估数据量、索引、执行计划、并发量和目标库压力。
2. 禁止在线接口做无上限全表查询、深分页 OFFSET、SELECT *、无索引排序/聚合/JOIN、一次性加载超大 ID 集合。
3. 查询必须按时间窗口、主键游标、分片或状态分页推进，并控制 batch_size、最大批次数、超时、并发和 sleep/backoff。
4. 大聚合、历史回补、跨库对账、批量修复和导出应走受控异步任务、维护队列、汇总表、ClickHouse 或数仓，不放在 HTTP 请求内长时间执行。
5. 批量写入和删除必须小批次按主键或索引推进，避免长事务、大锁、超大 IN/NOT IN；任务要支持幂等、断点续跑、进度水位和可中止。
6. 新增大数据查询必须补索引依据、保护阈值、关键日志/指标和最小测试；无法验证执行计划时，要说明风险和上线观察方式。

九、接口与前端
1. 前端联动字段命名以新版接口契约为准，字段别名必须在接口文档中明确声明。
2. 前端文案必须支持中文和英文国际化，不能硬编码业务文案。
3. 敏感操作必须接入登录态、签名、加密、秘钥版本或二次校验等既有安全机制。
4. 修改 security.secret_key、签名、加密或秘钥版本选路时，必须同步启动校验、接口文档、配置样例和最小单测。
5. 新增业务码必须补中文和英文文案；新增接口必须补接口文档。
6. 前端请求不得默认把整份表单、筛选条件或大 JSON 对象纳入签名/加密；只传后端接口文档 `### 安全字段` 表声明的关键字段。

十、验证与交付
1. 后端改动必须执行 gofmt；大范围改动执行 go test ./...，小范围至少跑相关包测试。
2. 前端改动至少执行 pnpm -F @vben/web-antd run typecheck；大范围改动优先执行 pnpm run check。
3. 存在 Makefile、CI、lefthook 或 pre-commit 时，优先执行项目已有检查入口。
4. 检查失败必须继续修复，不能绕过或只解释。
5. 最终说明必须包含：修改文件、改动原因、执行检查、未通过项、是否需要数据迁移或补偿任务。
```
