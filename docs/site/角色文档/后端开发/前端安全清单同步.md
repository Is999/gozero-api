# 前端安全清单同步

本文档约束 `gozero-api` 前台工程同步签名和加密策略的方式。当前工作区没有独立前台工程时，先以 `docs/site/route_security_manifest.json` 作为稳定同步入口。

## 同步入口

- 后端源头：`internal/handler/route_security_manifest.go:DefaultRouteSecurityManifest()`。
- 前端快照：`docs/site/route_security_manifest.json`。
- 漂移测试：`go test ./internal/handler` 会校验快照与后端清单一致。

## 前端策略

- 只按清单里的 `alias`、`method`、`path` 匹配路由；未知路由返回空安全策略。
- `chain=none` 的路由不参与前台签名、加密或登录态安全链路。
- `chain=internal` 的路由只允许内网运维调用，不应被公网前台页面直接调用。
- `requestSign`、`requestCipher`、`responseSign`、`responseCipher` 必须逐字段同步，不得自行扩展为 `*` 或 `cipher`。
- 请求签名和加密只覆盖清单声明的重要字段；大表单、大对象、分页列表和统计集合不得默认进入安全链路。

## 修改流程

1. 后端新增或修改路由时，先更新 `DefaultRouteContracts`、`RouteMeta`、`DefaultRouteSecurityContracts` 和 `RouteSecurityPolicies`。
2. 同步更新接口文档 `### 安全字段` 表。
3. 更新 `docs/site/route_security_manifest.json`。
4. 前台工程按快照同步 route policy 和路径匹配规则。
5. 后端执行 `go test ./internal/handler`，前端执行安全策略单测和 typecheck。

## 排障提示

- 前端请求缺少签名时，先确认目标路由是否在清单中且 `requestSign` 非空。
- 响应未验签时，先确认响应头包含 `X-Signature` 与 `X-Trace-Id`，且清单 `responseSign` 非空。
- 解密失败时，先核对 `X-Cipher` 是否只包含 `requestCipher` 或 `responseCipher` 声明的字段。
- 出现 `security_payload_too_large` 时，优先减少参与签名或加密字段，不要简单扩大上限。
