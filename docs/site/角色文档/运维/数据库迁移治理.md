# 数据库迁移治理

## 命令入口

迁移工具随主程序一起构建：

```bash
make build-tools
```

常用命令：

```bash
make migrate-status MIGRATE_CONFIG=./etc/config.yaml
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
make integration-test
```

`migrate-status` 和 `migrate-dry-run` 只读取 `schema_migrations` 并输出计划；`migrate-up` 会创建版本表、执行未登记迁移并写入 `version/name/asset/checksum/applied_at`。

## 当前基线

当前基线迁移：

| version | name | asset |
| --- | --- | --- |
| `202606050001` | `create_api_user` | `api_user_schema.sql.tmpl` |
| `202606050002` | `create_sys_config` | `sys_config_schema.sql.tmpl` |

`sys_config` 保持不加前缀，供 admin/API 共用运行期配置。后续新增前台业务表默认使用 `api_` 前缀。

## 发布要求

上线前必须完成：

1. `make migrate-dry-run MIGRATE_CONFIG=...` 确认计划。
2. 核对 `checksum`，历史版本不得被修改。
3. 执行后核对 `schema_migrations` 登记版本。
4. 如新增 destructive 或 bootstrap-only 迁移，必须显式标记并单独准备回滚或补偿方案。

迁移执行器会按分号拆分 SQL 并跳过文件内 `BEGIN/COMMIT`，不要求 MySQL DSN 开启 `multiStatements=true`。

`make integration-test` 会使用 `deploy/integration/docker-compose.yml` 启动 MySQL，并在真实 MySQL 上校验执行、幂等和 checksum 防篡改逻辑。
