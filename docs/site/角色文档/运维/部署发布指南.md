# 部署发布指南

## 目标

本文约束 `api` 前台 API 的上线、配置、数据库版本登记和生产观测流程。后台管理和运营任务由 `admin` 承担。

## 部署前检查

上线前确认：

1. 已执行 `make ci`，或至少完成 `gofmt`、`go test ./...`、`go build ./cmd/api`、`go build ./cmd/migrate`。
2. 已通过 `make migrate-dry-run MIGRATE_CONFIG=...` 确认数据库迁移计划。
3. `sys_config` 表使用未加前缀的表名；前台业务表使用 `api_` 前缀。
4. `etc/config.yaml` 已从环境模板生成，并配置强随机 `app_id`、`app_key`、`jwt_secret`。
5. 生产 `Mode=pro/prod/production` 已启用登录限流；开放注册时已启用注册限流。
6. 已配置 `ops.config_reload_token`，并确认 `/internal/system/...` 仅允许内网来源和正确 `X-Ops-Token`。
7. `/api/live`、`/api/ready`、`/api/metrics` 在目标环境按内网或网关白名单可访问。
8. `docs/site/route_security_manifest.json` 与前端安全清单同步，未知路由不默认进入全量签名或整包加密。

## 构建

```bash
make ci
make package VERSION=20260605
```

没有 `make` 的环境可等价执行：

```bash
gofmt -w $(find . -name '*.go' -not -path './vendor/*')
go test ./...
go build -o bin/api ./cmd/api
go build -o bin/api-migrate ./cmd/migrate
```

发布产物至少包含：

- `bin/api`
- `bin/api-migrate`
- `dist/api-<version>.tar.gz`
- `etc/config.yaml`
- 必要的证书、密钥和运行期配置文件

仓库提供的部署模板：

- `deploy/docker/Dockerfile`
- `deploy/systemd/api.service`
- `deploy/integration/docker-compose.yml`

## 数据库版本治理

上线工具应先执行 dry-run，再执行 up：

```bash
make migrate-dry-run MIGRATE_CONFIG=./etc/config.yaml
make migrate-up MIGRATE_CONFIG=./etc/config.yaml
```

当前基线迁移包括：

| version | name | asset |
| --- | --- | --- |
| `202606050001` | `create_api_user` | `api_user_schema.sql.tmpl` |
| `202606050002` | `create_sys_config` | `sys_config_schema.sql.tmpl` |

运维约束：

- 新环境按 `DefaultMigrations()` 顺序执行未登记版本，并写入 `version`、`name`、`asset`、`checksum`、`applied_at`。
- 已登记版本的 SQL 内容不得修改；需要变更时新增更高版本迁移。
- 发布前运行 `go test ./internal/database` 校验迁移清单和 checksum。
- `sys_config` 是 admin/API 共用运行期配置表，保持不加前缀。

详细规则见 [数据库迁移治理](数据库迁移治理.md)。

## 启动期硬校验

服务启动会在进入运行态前校验以下风险：

- `jwt_secret` 长度过短。
- Redis 地址为空或连接池大小无效。
- 认证密码最小长度低于安全下限。
- 登录/注册限流窗口、次数或锁定时间超过可控范围。
- Collector Redis 载体缺少 stream。
- 运维白名单配置公网 IP/CIDR。
- 生产环境使用占位密钥、未配置运维令牌、关闭登录限流或跳过 Redis TLS 证书校验。

## 运行

```bash
./bin/api -f ./etc/config.yaml
```

配置热加载只刷新运行期配置，不重建 HTTP 监听、MySQL、Redis、OTLP 等启动期连接。启动期配置变化会在热加载状态中标记为需重启。

## 观测与告警

发布后至少确认：

- `/api/live` 返回成功。
- `/api/ready` 中 MySQL、Redis、Collector 等组件状态正常。
- `/api/metrics` 可被 Prometheus 从内网抓取，公网不可直接访问。
- 日志中可通过 `trace_id`、`route`、`uid`、`app_id` 关联一次请求。
- 认证安全指标 `api_auth_security_events_total{app_id,action,reason,category}` 有基线数据。
- `category="security_config"` 出现即告警，优先排查 `security.secret_key`、`sys_config`、Redis 缓存和 redsync 回源。
- Prometheus 可加载 `docs/prometheus/api-alerts.yml`，并按生产基线调整阈值。
- Grafana 可导入 `docs/grafana/api-dashboard.json` 观察认证安全和 Collector 基线。

详细 PromQL 参考 [认证安全指标与告警](../后端开发/认证安全指标与告警.md)。

## 发布流程

1. 合并代码并完成测试。
2. 执行数据库迁移，确认 `schema_migrations` 中新版本和 checksum 登记成功。
3. 发布新二进制和配置。
4. 确认 `/api/ready` 正常。
5. 对登录、刷新 token、用户信息、系统配置读取执行低风险冒烟。
6. 若启用签名或加密，使用前端安全清单覆盖至少一个签名接口和一个加密接口。
7. 观察 10 到 30 分钟认证安全指标、错误日志和 Redis session 写入。

## 回滚

1. 摘除异常实例或回退流量。
2. 恢复上一版本二进制和配置。
3. 确认 `/api/ready`、登录、刷新 token 和 Redis session 正常。
4. 如果本次包含数据库迁移，按迁移单确认是否需要反向脚本或补偿任务。
5. 观察 `security_config`、`token`、`auth` 分类指标，确认异常恢复到基线。
