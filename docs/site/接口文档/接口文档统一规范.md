# 接口文档统一规范

本文档定义 `gozero-api` 前台 API 文档的统一书写样板。所有接口按“请求示例、字段说明、响应示例、响应字段说明、data 内部字段”的结构维护。

> **AI 开发必读**：AI 新增或修改接口文档前，必须先阅读 [AI开发规范](../角色文档/后端开发/AI开发规范.md)，并同步检查业务码、多语言文案、路由元数据和测试。

## 通用响应结构

### 响应示例

```json
{
  "status": true,
  "code": 1,
  "message": "操作成功",
  "data": {},
  "traceId": "0af7651916cd43dd8448eb211c80319c",
  "spanId": "b7ad6b7169203331"
}
```

### 响应字段说明

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| status | bool | 是 | false | 状态: true 成功, false 失败 |
| code | int | 是 | 0 | 业务状态码，不同业务错误返回不同状态码 |
| message | string | 否 |  | 当前语言下的响应消息，受 `Accept-Language` 影响 |
| data | object/array/null | 否 |  | 业务响应数据 |
| traceId | string | 否 |  | 链路追踪 ID，便于日志检索 |
| spanId | string | 否 |  | 当前服务 span ID |

## 通用请求头

| 字段 | 类型 | 是否必须 | 默认值 | 字段解释 |
|------|------|----------|--------|----------|
| Authorization | string | 条件必填 |  | 登录态接口必填，格式 `Bearer <token>` |
| Content-Type | string | 否 | application/json | JSON 请求体建议固定为 `application/json` |
| Accept-Language | string | 否 | zh-CN | 国际化语言，支持中文和英文 |
| X-Trace-Id | string | 否 |  | 请求链路追踪 ID，建议前端生成并透传 |
| X-App-Id | string | 条件必填 |  | 配置 `security.secret_key` 后必填，值为 base64 后的 `app_id` |
| X-Signature | string | 否 | R | 签名方式：R/RSA、A/AES、M/MD5 |
| X-Crypto | string | 否 | A | 加解密方式：A/AES、R/RSA |
| X-Cipher | string | 条件必填 |  | 请求加密字段列表，禁止使用 `cipher` 整包加密 |
| X-Key-Version | string | 否 |  | 指定秘钥版本；为空时按稳定/灰度配置选路，最长 64 位且不能包含空白或通配符 |
| X-Gray-Key | string | 否 |  | 灰度秘钥分桶键，通常使用用户 ID 或设备 ID |

## 签名与加密字段

- 接口文档必须列出参与请求签名、响应签名、请求加密和响应加密的具体字段；没有敏感字段时写明“不参与签名/加密”。
- 请求和响应签名禁止默认使用 `*` 对完整请求体、完整响应 `data`、大结构体或分页列表整体签名。
- 请求和响应加密禁止默认使用 `cipher` 整包加密，只允许声明字段级路径。
- 签名和加密优先覆盖 token、金额、订单号、手机号、邮箱、二次校验字段等重要数据，避免拖慢大响应解析。
- 单个接口参与签名或加密的字段最多 8 个；普通安全字段最大 4KB，`json:` 小对象最大 8KB。
- 安全中间件读取请求体最多 64KB；更大的业务负载应走专用上传、异步任务或拆分字段。

## 安全失败业务码

安全链路失败只返回脱敏后的业务码和多语言文案，真实错误进入日志和 trace。前端按 `code/message` 展示，不硬编码后端内部错误。

| 业务码 | 中文文案 | 场景 |
|--------|----------|------|
| 20006 | 安全 AppID 无效 | `X-App-Id` 缺失或格式错误 |
| 20007 | 安全秘钥不可用 | 秘钥配置、版本或缓存不可用 |
| 20008 | 请求签名校验失败 | 缺少签名、签名错误或签名方式不合法 |
| 20009 | 安全字段超过限制 | 请求体、字段数量、字段长度或复杂结构超过轻量处理上限 |
| 20010 | 加解密链路未启用 | 客户端声明密文但当前应用关闭加解密 |
| 20011 | 请求解密失败 | 请求加密字段声明、解密或明文校验失败 |
| 20012 | 响应签名处理失败 | 响应回签字段声明或签名生成失败 |
| 20013 | 响应加密处理失败 | 响应加密字段声明、编码或加密失败 |

### 安全字段表样板

```markdown
### 安全字段

| 项目 | 字段 |
|------|------|
| 请求签名字段 | username, password |
| 请求加密字段 | password |
| 响应签名字段 | token, expiresAt |
| 响应加密字段 | token, user.email |
```

## 链路追踪与排障

- 后端会将 `trace_id`、`span_id`、`uid`、`route`、`node` 写入访问日志和错误日志。
- 同一错误链路只在 handler、中间件或健康检查顶层打印一次 error 日志。
- 多实例部署时登录态保存在 Redis，单个 session 使用 String，用户 jti 索引使用 ZSET；排障时可结合 `uid`、`jti` 和 `trace_id` 查询。
- Redis session TTL 不超过 JWT 过期时间；鉴权成功会 best-effort 懒补用户 jti 索引，用户级全量失效通过索引精确删除，不扫描 Redis。
- 注册、登录、刷新、退出、登录态鉴权失败、限流和用户级 session 失效会 best-effort 投递 `auth.security` 认证风控事件；payload 只保留脱敏哈希和链路字段，默认 Processor 会统计 `gozero_api_auth_security_events_total{app_id,action,reason,category}`。
